# gateway — single entrypoint for the cinema stack

## Purpose

The gateway is a single `nginx:stable-alpine` container that holds **the only host port** in the entire cinema stack (`80`). Everything else — auth, admin panels, catalog, ugc, billing — listens only on the internal `cinema_net` and is reached by URL prefix through this nginx.

It also serves Django static and media files directly from named volumes, so admin requests don't proxy through gunicorn for assets.

## Quick start

```bash
docker network create cinema_net   # one-time, idempotent
docker compose up -d
```

Verify:

```bash
curl http://localhost/             # → plain-text landing page with the route list
```

The gateway is intentionally tolerant of missing upstreams. If a downstream service is not running, the corresponding prefix will return `502 Bad Gateway`, but the gateway itself stays up — you can bring stacks up and down independently behind it.

## Routing table

All paths are public-facing URLs (i.e. `http://localhost/<path>`). Upstreams are resolved by Docker DNS through the `cinema_net` network.

| Public prefix | Upstream | Path rewrite | Notes |
|---|---|---|---|
| `/auth/` | `auth-api:80` | `/auth/foo` → `/foo` | FastAPI started with `root_path=/auth` |
| `/notify/` | `notify-api:8001` | `/notify/foo` → `/foo` | FastAPI with `root_path=/notify` |
| `/admin/movies/` | `movies-admin:8000` | `/admin/movies/foo` → `/foo` | Django; assets handled below |
| `/admin/movies/static/` | (named volume `movies_static_volume`) | served directly by nginx | `alias`, `expires 7d`, access log off |
| `/admin/movies/media/` | (named volume `movies_media_volume`) | served directly by nginx | same |
| `/admin/notifications/` | `django-admin:8000` | `/admin/notifications/foo` → `/foo` | Django |
| `/admin/notifications/static/` | (named volume `notifications_static_volume`) | served directly by nginx | |
| `/admin/notifications/media/` | (named volume `notifications_media_volume`) | served directly by nginx | |
| `/content/` | `movies-api:8000` | `/content/foo` → `/foo` | catalog read API; FastAPI with `root_path=/content` |
| `/ugc/` | `ugc-content-api:8000` | `/ugc/foo` → `/foo` | likes / reviews / bookmarks |
| `/ugc-events/` | `ugc-api:8000` | `/ugc-events/foo` → `/foo` | click / page-view events |
| `/billing/` | `billing-api:8001` | `/billing/foo` → `/foo` | subscriptions / charges |
| `/` | — | — | landing page (text/plain) listing the routes |

Headers added to every proxied request: `Host`, `X-Real-IP`, `X-Forwarded-For`, `X-Forwarded-Proto`, `X-Request-Id`. The `X-Request-Id` is generated by nginx if the client didn't send one — used by every downstream service for trace correlation.

## Adding a new service behind the gateway

Say you're adding a new service `foo-api` that should answer on `/foo/*`.

1. **In the service's compose file:**
   - Give the container a stable `container_name: foo-api` so nginx can resolve it.
   - Join the external network `cinema_net` in the `networks:` block.
   - Drop any `ports:` host-binding — the gateway is the only public entrypoint.

2. **In the service's FastAPI (or framework) init:**
   - Pass `root_path="/foo"` (or the framework equivalent). This makes Swagger UI, OpenAPI links, and any reverse-URL helpers work correctly through the prefix.
   - If it's Django, set `FORCE_SCRIPT_NAME=/foo/` and skip the rewrite below (Django strips the prefix itself).

3. **In this gateway's `nginx.conf`:** add the block before the landing `location = /`:

   ```nginx
   location ^~ /foo/ {
       set $upstream_foo foo-api:8000;
       rewrite ^/foo/(.*)$ /$1 break;
       proxy_pass http://$upstream_foo;
   }
   ```

   The `set $upstream_foo ...` indirection is on purpose: nginx will lazy-resolve the hostname at request time via Docker's embedded DNS, so the gateway can start even when `foo-api` is not yet running. Without the variable, nginx would refuse to start until the upstream resolves.

4. **Restart the gateway** to pick up the config:

   ```bash
   docker compose restart gateway
   ```

5. **Add the new route to the landing page** (the `location = /` block at the bottom of `nginx.conf`).

6. **Update this README's routing table.**

## Integration with the cinema stack

- **Networks.** The gateway only joins `cinema_net`. Every other service must also join `cinema_net` to be reachable from here.

- **Static / media volumes.** Django admin assets are served by nginx itself (not by the Django container) via four named volumes, all declared `external: true`:

  | Volume | Owned by | Mounted at |
  |---|---|---|
  | `movies_static_volume` | `admin-panel/` | `/opt/admin/movies/static` |
  | `movies_media_volume` | `admin-panel/` | `/opt/admin/movies/media` |
  | `notifications_static_volume` | `notifications/` | `/opt/admin/notifications/static` |
  | `notifications_media_volume` | `notifications/` | `/opt/admin/notifications/media` |

  They must exist before `docker compose up -d` here. Starting the owning stack at least once creates them; after that they persist across gateway restarts.

- **Bring-up order.** Gateway can be started first or last — it doesn't care. Routes for not-yet-running services return `502`. The recommended flow from `cinema/`:

  ```bash
  make network    # cinema_net
  make auth       # so /auth works
  make admin      # so /admin/movies works
  ... etc
  make gateway    # bring up the entrypoint
  ```

- **Resource budget.** `deploy.resources.limits` caps the container at 32mb RAM and 0.2 CPU — nginx is happy in that envelope for the load of a pet stand.

- **Host port `80`** is the only port exposed by any cinema container. To run on a different host port, change the left side of the `ports:` mapping (`"8080:80"` etc.); the route URLs will become `http://localhost:8080/auth/`, etc.

## Logs

```bash
docker compose logs -f gateway     # nginx access + error log to stdout
```

Access log format (named `gw`):
```
[$time_local] $remote_addr "$request" -> $upstream_addr status=$status bytes=$body_bytes_sent rt=$request_time req_id=$request_id
```

`req_id` is the same `X-Request-Id` that downstream services log and that Jaeger uses for trace correlation, so you can grep across all services with one ID.
